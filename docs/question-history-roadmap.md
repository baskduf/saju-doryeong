# 질문 기록 확장 계획

이 문서는 `/fortune/[id]`의 `기록` 탭을 어디까지 확장할지에 대한 현재 합의안을 정리한다.

## 현재 상태

- 상세 운세 페이지의 `기록` 탭은 최근 질문 기록을 읽기 전용으로 보여준다.
- 현재 노출 수는 최신순 `10개`로 제한한다.
- `sample-user` 경로는 DB 대신 샘플 질문 기록을 보여준다.
- 질문 기록 저장은 `FortuneQuestionHistory` 테이블을 사용한다.

관련 구현 파일:

- `app/fortune/[id]/FortuneSections.tsx`
- `app/fortune/[id]/page.tsx`
- `lib/fortune-question-history.ts`
- `prisma/schema.prisma`

## 의사결정

질문 기록의 최종 구조는 아래 방향을 기본안으로 둔다.

- 상세 페이지의 `기록` 탭은 `최근 10개 미리보기` 역할만 맡는다.
- 전체 질문 기록 탐색은 `별도 페이지`로 분리한다.

즉, 가장 권장하는 구조는 아래와 같다.

1. `/fortune/[id]`
2. `기록` 탭에서 최근 10개 노출
3. 기록이 10개를 넘으면 `전체 기록 보기` 버튼 노출
4. `/fortune/[id]/history` 같은 전용 페이지에서 전체 기록 탐색

## 왜 이 방식이 좋은가

- 상세 페이지가 과도하게 길어지지 않는다.
- 운세/분석/기록의 역할이 섞이지 않는다.
- 기록이 많아져도 성능과 가독성을 유지하기 쉽다.
- 이후 `더보기`, `월별 묶음`, `검색`, `질문 다시 보기` 같은 기능을 붙이기 쉽다.
- 대부분 사용자는 최근 기록만 빠르게 보길 원하고, 과거 기록을 깊게 보는 사용자는 별도 공간으로 이동하는 편이 자연스럽다.

## 비권장 방향

아래 구조는 현재 기준으로 우선순위를 낮게 둔다.

- 상세 페이지 안에서 전체 기록을 모두 노출
- `기록` 탭 안에서 무한 스크롤만으로 확장
- 처음부터 검색, 필터, 페이지네이션을 모두 넣는 방식

이 방식들은 v1에서는 정보량과 복잡도를 빠르게 키우고, `/fortune/[id]`의 성격을 흐릴 가능성이 크다.

## 구현 원칙

### 1. 상세 페이지는 미리보기

- `기록` 탭에서는 최근 10개만 보여준다.
- 카드에는 아래 정보만 유지한다.
- 질문
- 판단 신호 2개
- 답변 제목
- 답변 본문 요약
- 참고 정책 문구
- 생성 시각

### 2. 전체 기록은 전용 페이지

- 전용 페이지는 `최근순` 기본 정렬을 유지한다.
- 첫 버전은 `cursor pagination` 또는 `다음 20개 보기` 정도의 단순한 방식으로 시작한다.
- 월별 그룹, 검색, 필터는 2차 확장으로 미룬다.

### 3. 저장 구조는 유지

- `FortuneQuestionHistory` 스키마는 그대로 사용한다.
- 상세 페이지는 `take 10` 조회만 담당한다.
- 전체 기록 페이지에서만 추가 조회 로직을 둔다.

## 단계별 로드맵

### 단계 1. 현재 상태 유지

- `/fortune/[id]`의 `기록` 탭은 최근 10개만 보여준다.
- `sample-user`는 샘플 기록으로 동작한다.

### 단계 2. 전체 기록 진입점 추가

- 기록이 10개를 넘는 경우에만 `전체 기록 보기` 버튼을 노출한다.
- 버튼은 `/fortune/[id]/history`로 이동한다.

### 단계 3. 전용 기록 페이지 추가

- `/fortune/[id]/history` 페이지를 만든다.
- 기본 기능은 아래 3개만 포함한다.
- 최근순 목록
- 다음 기록 더 보기
- 기록 1개 다시 읽기

### 단계 4. 선택적 고도화

- 월별 그룹
- 신호별 필터
- 질문 문장 검색
- 질문 재사용 또는 다시 묻기

## 비목표

현재 계획에는 아래를 포함하지 않는다.

- 질문 기록 수정
- 질문 기록 삭제
- 질문 기록 공유 전용 기능
- 관리자 로그와 질문 기록의 통합

## 결론

현재 서비스 단계에서 가장 좋은 구조는 `상세 페이지 최근 10개 미리보기 + 전체 기록 별도 페이지`다.

이 방향은 지금의 단순함을 유지하면서도, 기록이 많아졌을 때 자연스럽게 확장할 수 있는 가장 안정적인 방식으로 본다.
