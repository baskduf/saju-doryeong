# 사주 로직 계층 구조 정리

이 문서는 현재 서비스의 사주 로직을 `파일 순서`가 아니라 `해석 계층` 기준으로 설명한다.
핵심 목적은 다음 두 가지다.

- 사주 결과가 어떤 층을 거쳐 최종 문장으로 나오는지 한눈에 이해하기
- 명리 해석의 원천 층과 서비스 표현 층이 어디서 갈리는지 분명히 하기

## 1. 한눈에 보는 전체 흐름

현재 운세 시스템은 아래 순서로 움직인다.

1. 입력과 확정성 판단
2. 명식 생성
3. 원국 해석
4. 오늘 흐름 접속
5. 점수와 기본 운세 생성
6. 보조 체계 결합
7. signal 추상화
8. eventOutlook 사건 가능성 해석
9. 문장화와 소비처 전달

중요한 점은 `signal`과 `eventOutlook`가 해석의 출발점이 아니라는 것이다.
현재 시스템의 기준 순서는 끝까지 아래 원칙을 따른다.

- `명식 -> 원국 해석 -> 일진 보정 -> 기본 운세`
- `기본 운세 -> signal / eventOutlook -> UI / 질문 / 공유 / 카카오`

즉, 표현 계층이 명리 해석을 대신하면 안 되고, 어디까지나 번역 계층으로만 작동해야 한다.

## 2. 계층별 설명

### 2.1 입력과 확정성 계층

이 계층은 사용자가 준 생년월일, 출생시, 달력 기준, 조회 날짜를 받아 `정확 계산이 가능한지`를 먼저 판정한다.

- 입력
  - 생년월일
  - 출생시
  - 양력/음력/모름
  - 오늘 기준 날짜
- 여기서 결정되는 것
  - exact 계산 가능 여부
  - reference-only 경로 여부
  - 출생시 미상 시 정오 fallback 사용 여부

현재 시스템에서 가장 중요한 분기점은 `calendarType = unknown`이다.

- 양력/음력이 확정되면 exact 계층으로 내려간다.
- 모름이면 정확한 만세력 확정 대신 `solar-lunar-blend` 공통분모 reference 계층으로 간다.

즉, 이 층은 사주를 읽기 전에 `지금 결과가 확정 풀이인지 참고 풀이인지`를 먼저 정의하는 층이다.

### 2.2 명식 생성 계층

이 계층은 실제 만세력과 팔자를 만든다.

- 사주 팔자
  - 연주
  - 월주
  - 일주
  - 시주
- 함께 만들어지는 기초 정보
  - 일간
  - 오행 분포
  - 지장간
  - 납음
  - 태원/명궁/신궁/공망 같은 보조 정보

이 계층의 결과는 `명식 원본`이다.
아직 좋고 나쁨을 말하는 단계가 아니라, 해석을 위한 재료를 정리하는 단계다.

### 2.3 원국 해석 계층

이 계층은 명식을 바탕으로 `사람의 기본 구조`를 읽는다.

여기서 다루는 핵심 축은 다음과 같다.

- 일간 강약
- 월령과 계절력
- 통근 정도
- 지지 관계
- 십신 우세
- 격국 후보
- 용신/희신/기신/구신

현재 서비스에서 이 층은 가장 명리학적인 중심축이다.
최종 문장보다 훨씬 앞단에서 이미 아래 내용이 정해진다.

- 기본 성향 요약
- 균형이 필요한 오행
- 도와주는 기운과 부담이 되는 기운
- 원국 안의 충/형/합에 따른 구조적 압력

즉, 이 층은 `이 사람의 기본 판이 어떻게 생겼는가`를 정하는 층이다.

### 2.4 오늘 흐름 접속 계층

이 계층은 원국 위에 `오늘의 간지`를 붙인다.

주요 계산 축은 다음과 같다.

- 오늘 천간/지지
- 오늘 일진과 일간의 오행 관계
- 오늘 지지와 원국 지지의 상호작용
- 오늘 오행이 용신/기신에 얼마나 맞는지
- 오늘 기세가 확장 쪽인지 압박 쪽인지

현재 서비스에서 이 계층은 정적 사주를 동적 운세로 바꾸는 핵심이다.
즉, 원국이 뼈대라면 이 층이 그날의 기류를 얹는다.

결과적으로 여기서 아래 판단이 생긴다.

- 오늘 관계가 비겁/식상/재성/관성/인성 중 어디에 가까운지
- 지지 상호작용이 부드러운지 거친지
- 용신 적중도가 올라오는지, 기신 압박이 올라오는지
- 오늘은 미는 날인지, 늦추는 날인지

### 2.5 점수와 기본 운세 계층

이 계층은 앞선 명리 결과를 서비스가 다룰 수 있는 기본 운세 객체로 만든다.

여기서 생성되는 대표 결과는 다음과 같다.

- score
- grade
- headline
- summary
- detail
- caution
- recommendedActions
- avoidToday
- categoryScores
- luckyHints

중요한 점은 이 층까지가 아직 `사주 본문`이라는 것이다.
즉, 이 단계의 결과는 signal이나 카드용 UI가 아니라, 명리 판단을 서비스 문장으로 1차 번역한 기본 운세다.

### 2.6 보조 체계 결합 계층

현재 시스템은 사주만 단독으로 쓰지 않고 보조 체계를 붙인다.
다만 우선순위는 분명하다.

- 사주: 기본축
- 구성학: 보정축
- 육효: 질문축

구성학은 아래 역할을 맡는다.

- score 보정
- narrative tone 보정
- category 강조점 보정

즉, 사주를 뒤엎는 역할이 아니라 `조금 더 현실적인 강조점`을 주는 보조 레이어다.

육효는 기본 일일 운세를 다시 계산하지 않는다.
질문 응답에서만 들어오며 아래 역할을 맡는다.

- 질문 시점의 방향성 보강
- caution/action/timing 힌트 제공
- daily fortune와 충돌 시 충돌 메타를 남기며 조정

즉, 육효는 기본 운세 엔진이 아니라 `질문형 응답 보조 엔진`이다.

### 2.7 Signal 추상화 계층

이 계층은 기본 운세를 서비스 친화적인 공통 언어로 바꾼다.

현재 signal은 다음 7개 축으로 정리된다.

- momentum
- friction
- timing
- work
- money
- relationship
- recovery

이 층의 목적은 명리 해석을 다시 만드는 것이 아니다.
목적은 아래와 같다.

- UI 카드 구조를 통일하기
- 질문 응답이 같은 판단 축을 사용하게 만들기
- 공유 payload와 외부 소비처가 같은 언어를 쓰게 만들기

즉, signal은 `해석의 원천`이 아니라 `공통 표현 레이어`다.

여기서 중요한 표현 원칙도 하나 있다.

- `timing`은 항상 시간대 예언처럼 말하지 않는다.
- 시간대 표현은 timing이 실제로 강한 날에만 전면에 둔다.
- 평소에는 `초반에 붙는 흐름`, `후반에 반응이 커지는 흐름`, `서두르면 놓치기 쉬운 순서`처럼 사건 흐름 중심으로 번역한다.

### 2.8 EventOutlook 사건 가능성 계층

이 계층은 signal과 오늘 보정값을 다시 묶어서 `오늘 어떤 사건 형태가 드러나기 쉬운가`를 만든다.

현재 공개 구조는 아래만 유지된다.

- kind
- intensity
- confidenceMode
- lead
- reason
- basisSignals

하지만 내부적으로는 일부 kind가 더 세분화되어 있다.

- conflict
  - branch-clash
  - directive-pressure
  - friction-overload
- contact
  - incoming-contact
  - relationship-realignment
- money-shift
  - expense-pressure
  - deal-opportunity
  - settlement-movement

이 계층의 의미는 중요하다.

- signal이 `무슨 축이 강한가`를 말한다면
- eventOutlook는 `그래서 어떤 사건 형태로 보이는가`를 말한다

즉, 기존의 조언형 문장을 조금 더 사건 가능성형으로 바꾸는 중간 해석 계층이다.

따라서 eventOutlook가 이미 사건 원인을 설명하고 있다면, signal이나 보조 문장은 그 말을 반복하기보다 이유와 실무 포지션을 보강하는 쪽이 맞다.

### 2.9 문장화와 소비처 계층

마지막 계층은 계산 결과를 실제 사용자 표면으로 내보낸다.

여기서 이루어지는 일은 다음과 같다.

- headline / summary / detail 구성
- recommendedActions / avoidToday 정리
- 질문 응답 생성
- 공유 payload 저장
- 카카오 카드/설명 생성
- 필요 시 LLM 내러티브 오버라이드

중요한 원칙은 이 층에서 새로운 사실을 만들지 않는다는 점이다.

- LLM은 해석을 발명하지 않는다.
- 이미 계산된 signal과 eventOutlook를 더 자연스러운 문장으로만 정리한다.
- 질문 응답도 daily fortune를 기본으로 두고 육효를 보강축으로만 사용한다.

즉, 이 층은 `표현층`이지 `재해석층`이 아니다.

## 3. 현재 시스템을 이해할 때 꼭 잡아야 할 축

### 3.1 exact와 reference는 다른 결과다

현재 시스템은 `불확실성을 숨기지 않는다`.

- exact
  - 만세력 확정
  - 일진과 원국의 직접 상호작용 반영
- reference
  - 양력/음력 공통분모 기준
  - 참고용 흐름
  - 확정 판정 어조 금지

즉, reference 결과는 exact의 하위 품질이 아니라 `다른 모드`다.

### 3.2 사주가 중심이고 signal은 번역층이다

현재 구조를 볼 때 가장 많이 헷갈리는 부분이 이것이다.

- 사주 해석이 먼저
- signal은 그 결과를 소비처 친화적으로 요약
- eventOutlook는 그 신호를 사건 가능성형으로 다시 정리

따라서 signal 점수나 카드 구조가 명리 해석 그 자체라고 보면 안 된다.

### 3.3 질문 응답은 daily fortune 단독 시스템이 아니다

질문 응답은 아래 3축을 함께 쓴다.

- daily fortune
- selected signals
- yukhyo reading

다만 우선순위는 여전히 `daily fortune 기반`이다.
육효는 질문 상황에서만 들어와 방향을 조정한다.

중요한 점은 육효가 질문 답변의 `첫 문장`을 먹지 않는다는 것이다.

- opening은 daily fortune / eventOutlook가 잡는다
- 육효는 이유, timing, caution을 보강한다
- 즉 육효는 질문형 응답의 보조축이지 별도 본체가 아니다

### 3.4 LLM은 후처리다

현재 LLM은 엔진을 대신하지 않는다.

- deterministic facts는 코드가 정한다
- LLM은 톤과 연결 문장을 다듬는다
- exact/reference, signal, eventOutlook의 기준은 바꾸면 안 된다

즉, LLM이 영감처럼 새 풀이를 내는 구조가 아니다.

## 4. 명리학 관점에서의 현재 위치

현재 시스템은 `정통 감정 엔진`이라기보다 `명리 문법을 서비스용으로 번역한 실용형 구조`에 가깝다.

강점은 다음과 같다.

- 원국과 일진의 구분이 분명하다
- exact/reference 구분이 정직하다
- 용신/기신, 강약, 격국 후보, 지지 상호작용을 기초축으로 유지한다
- signal과 eventOutlook가 명리 해석 위에 올라가도록 설계되어 있다

한계도 분명하다.

- 대운/세운/월운 같은 장기 운 계층은 얕다
- 격국 확정과 조후/통관 같은 고급 판정은 서비스형 근사치다
- eventOutlook subtype도 명리 판단의 완전한 정식 분류가 아니라 서비스용 원인축이다

즉, 현재 구조는 `명리 해석을 무시한 운세 생성기`는 아니지만, `전문 감정실 수준의 정통 명리 엔진`도 아니다.

## 5. 문서를 읽을 때의 실무 기준

새 기능을 붙일 때는 먼저 어느 계층의 변화인지부터 정하는 것이 좋다.

- 명식 계산을 바꾸는가
- 원국 해석을 바꾸는가
- 오늘 보정 규칙을 바꾸는가
- signal 추상화를 바꾸는가
- eventOutlook 사건 가능성을 바꾸는가
- 표현층만 바꾸는가

이 질문이 먼저 서면, 로직이 꼬이기 시작하는 지점을 훨씬 빨리 발견할 수 있다.

특히 아래 두 가지는 섞지 않는 편이 좋다.

- 명리 해석 변경
- 표현/카피 변경

현재 시스템은 층을 분리해 두었기 때문에, 어떤 수정이 해석 엔진 변경인지 단순 문장 변경인지 먼저 구분하는 것이 중요하다.
